# Память LightRAG на Render (данные после редеплоя)

Файловая система контейнера **временная**: без постоянного диска индекс и загрузки **пропадают** при новом деплое.

## Вариант: Persistent Disk (рекомендуется для файлового режима)

1. **Render** → сервис **LightRAG** → **Disks** → **Add Disk**.
2. Укажи размер (например **10 GB**) и **Mount path**, например **`/var/data`** (запомни путь — его нельзя потом менять без переноса данных).
3. **Environment** на том же сервисе добавь или измени (пути **внутри** mount path; префикс должен совпадать с **Mount path** диска, ниже для `/var/data`):

   | Variable | Value |
   |----------|--------|
   | `WORKING_DIR` | `/var/data/rag_storage` |
   | `INPUT_DIR` | `/var/data/inputs` |

   Если mount path другой (например `/mnt/data`), замени префикс: `/mnt/data/rag_storage` и `/mnt/data/inputs`.

4. **Save** → **Manual Deploy**.

Убедись, что **нет** старых значений `WORKING_DIR=./rag_storage` или путей вне диска — они перекроют шаблон и снова сломают персистентность.

Пока эти переменные указывают на каталог **на диске**, данные **переживают** редеплой. Каталоги создаст приложение при старте.

### Ограничения Render

- С диском **нет** zero-downtime deploy.
- **Один** инстанс (нельзя масштабировать горизонтально с одним диском).
- Сохраняется только то, что **под mount path**; `./rag_storage` в корне контейнера **не** сохраняется.

### Если раньше данные были «в контейнере»

После первого успешного запуска с диском каталоги на томе **пустые** — документы нужно **загрузить снова** (или восстановить из бэкапа, если делал).

## Вариант: внешняя БД

PostgreSQL / MongoDB и т.д. — см. `env.example` (хранилища `PG*`, `MONGO_*`). Данные живут в облачной БД, редеплой приложения их не стирает. Сложнее в настройке.

## Шаблон переменных

См. [env.render.example](../env.render.example) — блок `WORKING_DIR` / `INPUT_DIR`.
